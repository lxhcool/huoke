# Joinf / LinkedIn 登录态抓取方案设计

## 1. 目标

本方案用于说明如何在 `cloud.joinf.com` 无 API 的前提下，通过登录态和页面抓取方式获取：

- Joinf 商业数据
- Joinf 海关数据
- LinkedIn 公司与联系人增强信息

目标不是立即做全量高频采集，而是先建立一套稳定、可控、可扩展的采集链路。

## 2. 约束条件

当前已知约束：

- `cloud.joinf.com` 无开放 API
- 必须先登录才能进入数据营销模块
- 客户指定抓取入口为：
  - 顶部 `数据营销`
  - 其中的 `商业数据`
  - 其中的 `海关数据`
- LinkedIn 也希望补充抓取信息，但未明确要求首版大规模抓取

## 3. 推荐技术路线

建议采用：

> 浏览器自动化 + 会话复用 + 列表页抓取 + 详情页补抓 + 入库映射

推荐拆成以下模块：

- `登录管理模块`
- `Joinf 列表抓取模块`
- `Joinf 详情补抓模块`
- `LinkedIn 增强抓取模块`
- `字段映射与入库模块`
- `抓取任务调度模块`

## 4. Joinf 抓取流程

### 4.1 登录管理

建议方式：

- 使用浏览器自动化工具登录一次
- 成功后保存 cookie / session storage / local storage
- 后续任务优先复用登录态
- 登录失效时触发人工重新登录或登录更新流程

建议不要：

- 每次任务都重新输入账号密码
- 在代码中硬编码账号密码

### 4.2 页面进入路径

采集流程建议固定为：

1. 进入 `cloud.joinf.com`
2. 完成登录态校验
3. 点击顶部 `数据营销`
4. 进入 `商业数据` 或 `海关数据`
5. 根据关键词、国家、筛选条件发起查询
6. 抓取列表页结果
7. 对高价值记录进入详情页补字段

### 4.3 列表页抓取策略

列表页优先抓取：

- 公司名称
- 国家/地区
- 城市
- 行业
- 官网
- 联系人摘要
- 企业邮箱摘要
- 海关活跃摘要
- 列表项详情页链接

优先原因：

- 抓取效率高
- 足以先形成候选线索池
- 可以减少详情页访问次数

### 4.4 详情页补抓策略

详情页只对以下记录补抓：

- 排名前 N 的结果
- 符合批发小单偏好的结果
- 海关活跃度较高的结果
- 联系方式字段缺失但值得进一步补齐的结果

详情页建议补抓：

- 公司简介
- 联系人姓名与职位
- 电话与邮箱
- 员工规模
- 成立时间
- 标签信息
- 原始来源链接

## 5. LinkedIn 抓取策略

## 5.1 角色定位

LinkedIn 在本项目中建议定位为：

- `增强信息源`
- `公司画像补充源`
- `联系人验证源`

不建议在首版定位为：

- 主数据源
- 大规模全量采集源

## 5.2 推荐抓取范围

首版建议只抓：

- 公司主页
- 少量关键联系人主页

优先抓取字段：

- 公司简介
- 员工规模
- 联系人姓名
- 职位
- 个人主页链接

## 5.3 触发条件

只有当 Joinf 数据满足以下情况时，才触发 LinkedIn 增强抓取：

- 公司基础信息已明确
- 需要补联系人信息
- 需要验证岗位真实性
- 该公司分数较高，值得进一步增强

## 6. 抓取任务设计

建议任务拆分为三类：

- `搜索任务`
  - 输入关键词、国家、海关条件
  - 输出候选公司列表
- `详情补全任务`
  - 输入公司详情 URL
  - 输出完整公司和联系人信息
- `增强任务`
  - 输入公司名或公司主页信息
  - 输出 LinkedIn 补充画像

## 7. 存储与入库流程

抓取结果不要直接进主档，建议采用以下流程：

1. 原始结果落原始层
2. 执行字段映射
3. 进行标准化与去重
4. 建立公司/联系人/海关关系
5. 写入主档表
6. 记录来源追踪信息

## 8. 调度建议

### 8.1 手动触发阶段

首版建议先手动触发采集：

- 输入关键词后人工点击采集
- 观察页面结构稳定性
- 快速调整选择器和字段映射

### 8.2 半自动阶段

当抓取稳定后，可改成：

- 定时任务拉取
- 按国家/关键词批次运行
- 对高分结果做详情补抓

## 9. 稳定性建议

为了降低页面变化和登录态失效带来的问题，建议：

- 所有关键页面保留 DOM 选择器版本记录
- 对页面结构变动做失败告警
- 登录态失效时给出明确错误而不是静默失败
- 每个抓取批次保存截图或 HTML 快照

## 10. 风险与应对

### 10.1 Joinf 页面结构变更

应对：

- 采用稳定选择器优先
- 抽离页面适配层
- 保留抓取失败截图

### 10.2 登录态失效

应对：

- 会话复用
- 登录检测
- 失败后人工重新授权

### 10.3 LinkedIn 风控

应对：

- 降低频率
- 仅抓高价值结果
- 作为增强源而非主源

### 10.4 采集字段过多导致首版过重

应对：

- 字段标准按增强版定义
- 采集执行按 P0 / P1 分阶段落地

## 11. 第一阶段建议输出物

建议开发第一阶段先产出：

- Joinf 登录态管理脚本
- 商业数据列表页抓取脚本
- 海关数据列表页抓取脚本
- 原始结果落盘结构
- 字段映射器
- 导入当前系统的入库脚本

## 12. 下一步开发顺序

建议按下面顺序继续：

1. 补系统字段结构
2. 定义原始数据落盘格式
3. 编写 Joinf 登录态与列表页抓取器
4. 编写字段映射器
5. 接入主档入库逻辑
6. 再做 LinkedIn 增强抓取
