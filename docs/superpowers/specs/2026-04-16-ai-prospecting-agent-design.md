# AI 获客线索发现 Agent 设计文档

## 1. 文档信息

- 文档日期：2026-04-16
- 当前阶段：方案设计
- 使用场景：公司内部使用
- 目标用户：外贸业务员
- 文档目标：明确内部版 Agent 的业务边界、数据改造方向、首版能力与落地路径

## 2. 背景与问题定义

公司希望开发一个面向外贸业务员的 AI Agent，用于提升海外客户开发效率。当前已有一定规模的数据资产，且公司本身具备数据能力，但核心问题不在于“有没有数据”，而在于“数据未被整理成可稳定检索、可解释、可调用的线索引擎”。

当前问题主要集中在以下几个方面：

- 多个数据源分散在不同库、表、文件或系统中
- 缺乏统一的公司主档与联系人主档
- 海关数据、公司数据、联系人数据之间缺少稳定关联关系
- 数据字段命名不统一、质量不稳定、去重困难
- 业务员无法通过自然语言或统一界面快速找到“值得开发的目标客户”

因此，这个项目的本质不是先做一个会聊天的大模型，而是先建设一个可被 Agent 调用的获客线索引擎。

## 3. 项目定位

本项目首版定位为：

> 一个面向外贸业务员的内部线索发现 Agent。

它的职责不是一次性替代整个销售流程，而是优先解决两个最高频、最刚需的问题：

- 找目标客户公司
- 找联系人与企业邮箱

首版重点依赖两类输入条件：

- 产品关键词 + 国家/地区
- 海关进出口记录等采购活跃信号

## 4. 目标用户与使用方式

### 4.1 目标用户

- 一线外贸业务员

### 4.2 用户诉求

业务员希望通过尽可能少的输入，快速拿到一批可开发的目标客户名单，而不是自己跨多个平台手工搜索、筛选、验证和整理。

### 4.3 典型输入方式

业务员输入：

- 产品关键词，例如：`laser cutting machine`
- 国家/地区，例如：`Germany`、`USA`
- 可选过滤条件，例如：
  - 最近 12 个月有进口记录
  - 指定 HS Code
  - 公司规模区间
  - 目标岗位类型

### 4.4 典型输出结果

系统输出：

- 推荐公司名单
- 每家公司对应的联系人与企业邮箱
- 命中理由与排序原因
- 数据来源说明
- 海关记录或采购活跃度标签

## 5. 当前现状判断

基于现有沟通，当前数据状态更接近以下情况：

- 有较多现成数据资产
- 数据分散在不同库表
- 尚未形成统一主档
- 不同来源之间缺少可复用主键或统一实体映射

这意味着项目的第一难点不是 Agent 推理，而是数据层的可用性建设。若底层数据没有打通，即使 Agent 可以理解问题，也无法稳定返回可信结果。

## 6. 可行性判断

## 6.1 总体结论

该项目可行，且适合先在公司内部落地。

### 可行的原因

- 已有数据资产，具备形成壁垒的基础
- 目标用户明确，业务场景聚焦
- 首版聚焦在“找公司 + 找联系人”，价值直接、验证快
- 内部使用对多租户、权限体系、计费系统要求较低，落地阻力更小

### 主要难点

- 多源数据打通
- 公司与联系人统一主档建设
- 去重、标准化、实体映射
- 海关记录与公司实体关联
- 联系人和邮箱准确率控制

### 风险不在于大模型本身

真正影响首版效果的，不是模型是否足够聪明，而是：

- 数据是否可统一查询
- 返回结果是否可信
- 排序是否符合业务员真实需求

## 7. 方案路线对比

### 方案一：直接做统一检索助手

做法：直接连接现有多库，Agent 负责改写查询、调用多个检索接口、汇总结果给业务员。

优点：

- 开发速度快
- 容易快速出内部 Demo
- 能尽快验证业务员是否愿意使用

缺点：

- 结果稳定性差
- 跨源去重难
- 联系人映射不稳
- 后续能力扩展成本高

### 方案二：先建设统一主档，再接 Agent

做法：先建立公司主档、联系人主档和海关记录关联层，再让 Agent 在统一数据层上完成检索和解释。

优点：

- 结果稳定
- 排序和解释更可信
- 后续可扩展到邮件、CRM、评分等能力

缺点：

- 前期建设投入较大
- 首个可见成果产出速度相对较慢

### 方案三：直接做全自动获客 Agent

做法：让用户只输入产品和国家，系统自动完成找公司、找联系人、打分、写邮件、触达与跟进。

优点：

- 想象空间大
- 业务叙事强

缺点：

- 对当前数据基础要求过高
- 首版失败概率高
- 系统复杂度过高，不适合目前阶段

## 8. 推荐方案

推荐采用：

> 方案二为目标架构，方案一为启动方式。

即：

- 前台先做成可用的统一检索 Agent
- 后台同步建设公司主档、联系人主档和海关关联层

这样可以兼顾短期可见成果与长期可扩展性。

## 9. 首版产品定义

首版不是一个泛化聊天机器人，而是一个结构化的线索发现 Agent。

### 9.1 核心能力

- 理解业务员输入的自然语言检索意图
- 将自然语言改写为标准化查询条件
- 从多个数据源中检索候选公司
- 聚合联系人与企业邮箱
- 引入海关数据做采购活跃度筛选
- 对结果进行排序并给出推荐理由

### 9.2 典型查询示例

例如业务员输入：

`帮我找德国做钣金加工、最近一年有进口记录、最好能找到采购经理邮箱的公司`

Agent 输出：

- 公司列表
- 每家公司命中的关键词
- 海关记录摘要
- 联系人职位与邮箱
- 推荐优先级

## 10. 首版能力边界

### 10.1 必做能力

- 产品关键词扩展
- 国家与地区筛选
- 多数据源公司检索
- 公司去重与实体合并
- 联系人补全
- 企业邮箱聚合与可信度标记
- 海关记录关联与筛选
- 命中原因解释
- 搜索结果导出

### 10.2 暂不纳入首版

- 自动邮件发送
- LinkedIn 自动触达
- WhatsApp 自动化触达
- 多语言开发信生成
- 全自动无人值守 Agent
- CRM 深度双向同步

## 11. Agent 形态设计

首版建议采用“搜索工作台 + Agent 助手”的形态，而不是单纯聊天框。

### 11.1 前台交互建议

- 左侧：搜索条件区
- 中间：结果列表区
- 右侧：公司详情与联系人详情
- 顶部：自然语言输入框

### 11.2 Agent 的角色

Agent 不负责替代界面，而是负责：

- 帮用户理解和拆解查询
- 自动补全搜索条件
- 自动选择合适的数据源查询
- 对结果做解释与排序

## 12. 数据层设计方向

这是项目成功的关键部分。

### 12.1 建议建立的核心主档

- `company_master`：公司统一主档
- `contact_master`：联系人统一主档
- `customs_record`：海关进出口记录
- `company_contact_relation`：公司与联系人关系表
- `company_customs_relation`：公司与海关记录映射表
- `company_domain_relation`：公司与官网域名映射表
- `source_trace`：数据来源追踪表

### 12.2 公司主档建议字段

- 公司统一 ID
- 公司标准名称
- 别名/原始名称
- 国家
- 城市
- 官网域名
- 行业标签
- 公司规模
- 数据来源
- 数据可信度评分
- 最后更新时间

### 12.3 联系人主档建议字段

- 联系人统一 ID
- 姓名
- 职位
- 邮箱
- 邮箱类型
- 所属公司 ID
- LinkedIn 或其他来源链接
- 数据来源
- 可信度评分
- 最近更新时间

### 12.4 海关记录建议字段

- 海关记录 ID
- 进出口主体名称
- 关联公司 ID
- 交易时间
- HS Code
- 产品描述
- 金额或频次
- 贸易方向
- 数据来源

## 13. 关键数据治理任务

在正式让 Agent 大规模服务业务员之前，建议优先完成以下工作：

- 统一公司命名规则
- 统一国家、地区、行业字典
- 构建公司去重规则
- 构建联系人去重规则
- 处理公司与官网域名的映射
- 建立海关主体与公司主档的匹配规则
- 为所有结果保留来源与置信度

如果这些基础工作缺失，Agent 很容易出现以下问题：

- 重复推荐同一家公司
- 推荐错误联系人
- 海关记录与公司不匹配
- 无法解释“为什么推荐”

## 14. 首版检索流程

一个标准查询建议采用以下流程：

1. 用户输入自然语言查询
2. Agent 识别产品词、国家、海关筛选条件、联系人偏好
3. 查询改写模块生成标准化检索参数
4. 检索层调用多个数据源获取候选公司
5. 去重与实体合并模块对候选公司归并
6. 联系人补全模块补齐关键联系人与邮箱
7. 海关数据模块匹配对应进出口记录
8. 评分模块计算综合优先级
9. Agent 输出推荐结果与解释说明

## 15. 排序与评分建议

排序是首版体验的核心。

建议综合以下维度计算线索优先级：

- 产品关键词相关性
- 国家和地区匹配度
- 海关活跃度
- 最近交易时间
- 联系人完整度
- 企业邮箱完整度
- 公司规模匹配度
- 数据可信度

建议将分数拆成多个子分，便于业务员理解结果，例如：

- 关键词匹配分
- 海关活跃分
- 联系人完整分
- 综合推荐分

## 16. 成功指标建议

首版不建议直接把邮件回复率作为核心指标，因为首版重点不是触达，而是线索质量。

建议首版指标改为：

- 单次查询返回首批结果时间
- 查询结果中可用公司占比
- 查询结果中可用联系人占比
- 企业邮箱命中率
- 海关记录关联成功率
- 业务员导出/收藏使用率
- 业务员对推荐结果的采纳率

## 17. 分阶段落地建议

### 阶段一：内部原型验证

目标：先让业务员用起来。

输出：

- 一个统一搜索界面
- 一个自然语言输入入口
- 能返回公司、联系人、邮箱与海关摘要

### 阶段二：数据主档建设

目标：提升结果稳定性与可信度。

输出：

- 公司主档
- 联系人主档
- 去重规则
- 海关映射规则

### 阶段三：排序优化与推荐解释

目标：提升业务员对系统结果的信任度和使用频率。

输出：

- 评分模型
- 推荐理由解释模块
- 导出与收藏机制

### 阶段四：扩展后续能力

待前面稳定后，再考虑扩展：

- 邮件生成
- 邮件发送与跟进
- CRM 对接
- 多语言触达

## 18. 风险与应对

### 风险一：数据打不通

应对：先聚焦最关键的三类实体，公司、联系人、海关记录。

### 风险二：结果重复或错误

应对：先建设去重和统一主档，不急于扩展功能面。

### 风险三：业务员不信任结果

应对：所有推荐结果附带命中理由、来源和置信度。

### 风险四：首版范围过大

应对：严格限定在“找公司 + 找联系人 + 海关筛选”，不把邮件自动化纳入首版。

## 19. 结论

这个项目可以做，而且适合先在公司内部落地。

从现状看，最优路径不是直接做一个全自动销售 Agent，而是先做一个面向业务员的线索发现 Agent，并同步建设统一数据主档。

项目的关键成功因素不在于模型本身，而在于：

- 数据是否打通
- 主档是否稳定
- 排序是否可信
- 推荐结果是否可解释

首版一旦把“找目标公司 + 找联系人 + 海关筛选”这条链路跑通，就已经具备明确业务价值，并且为后续邮件生成、自动触达、CRM 集成打下基础。

## 20. 首版详细功能清单

本节将首版功能进一步拆到可评估、可排期、可验收的粒度。

### 20.1 查询输入模块

目标：让业务员用最低学习成本发起一次有效搜索。

功能点：

- 自然语言输入框
- 结构化筛选项
- 查询历史记录
- 常用搜索模板
- 查询条件保存

建议支持的结构化条件：

- 产品关键词
- 国家/地区
- 目标岗位
- 是否必须有企业邮箱
- 是否必须有关联海关记录
- 海关时间范围
- HS Code
- 公司规模
- 行业标签

首版验收标准：

- 用户可以通过自然语言或结构化表单发起查询
- Agent 能将自然语言解析为标准条件
- 用户可对解析结果进行人工确认和修改

### 20.2 关键词扩展模块

目标：把业务员输入的中文或简单英文产品词扩展成更适合检索的查询词。

功能点：

- 中文转英文行业词
- 产品同义词扩展
- 行业相关词扩展
- 词组归一化
- 查询词黑名单过滤

输出示例：

- 原始词：`激光切割机`
- 扩展词：`laser cutting machine`、`sheet metal laser cutter`、`industrial laser cutting equipment`

首版验收标准：

- 单个产品词至少输出 3 到 10 个可检索变体
- 可人工勾选保留或删除扩展词

### 20.3 公司检索模块

目标：从多个内部数据源中返回候选公司集合。

功能点：

- 多源并行检索
- 关键词召回
- 国家筛选
- 行业筛选
- 官网域名识别
- 候选公司合并

返回字段建议：

- 公司名
- 标准公司名
- 国家/城市
- 官网
- 行业
- 来源数
- 原始命中词
- 初始相关性分

首版验收标准：

- 支持同时查询多个数据源
- 支持返回来源明细
- 支持输出候选公司原始命中信息

### 20.4 公司去重与实体合并模块

目标：解决不同来源里同一家公司重复出现的问题。

功能点：

- 公司名称标准化
- 域名归一化
- 国家与城市辅助匹配
- 别名映射
- 人工确认合并

首版规则建议：

- 域名一致优先视为同一公司
- 标准化名称高相似且国家一致时合并
- 海关主体名与公司名高相似时进入候选映射池
- 保留人工修正入口，避免误合并

首版验收标准：

- 同一公司在结果页只展示一次主记录
- 可展开查看所有来源记录
- 合并逻辑具备可追溯性

### 20.5 联系人补全模块

目标：为目标公司找到优先可触达的人。

功能点：

- 联系人召回
- 目标岗位识别
- 联系人与公司关系映射
- 联系人去重
- 联系人可信度评分

岗位优先级建议：

- 采购经理
- 采购总监
- 供应链负责人
- CEO/Founder
- 销售采购双职能负责人

返回字段建议：

- 姓名
- 职位
- 邮箱
- 邮箱类型
- LinkedIn 或来源链接
- 联系人可信度分

首版验收标准：

- 每家公司至少能返回 0 到 N 个候选联系人
- 联系人结果带置信度和来源
- 用户可以手动指定目标岗位优先级

### 20.6 企业邮箱处理模块

目标：尽可能返回可信、可用的企业邮箱，而不是泛邮箱。

功能点：

- 邮箱格式校验
- 企业域名一致性校验
- 泛邮箱识别
- 同人多邮箱去重
- 邮箱可信度打分

可信度建议因素：

- 邮箱域名是否与官网一致
- 是否为企业邮箱而非免费邮箱
- 是否存在历史使用记录
- 是否与联系人姓名模式匹配

首版验收标准：

- 标记企业邮箱和免费邮箱
- 输出邮箱可信度等级
- 默认优先展示企业邮箱

### 20.7 海关数据筛选模块

目标：利用海关记录识别更有采购活跃度的目标公司。

功能点：

- 海关主体匹配
- HS Code 过滤
- 时间范围过滤
- 频次与金额统计
- 贸易活跃标签生成

建议输出标签：

- 最近 3 个月活跃
- 最近 12 个月持续进口
- 高匹配 HS Code
- 高频进口客户

首版验收标准：

- 能对公司展示最近时间段内海关活跃摘要
- 支持按海关条件筛选结果
- 支持展示匹配依据

### 20.8 排序与推荐解释模块

目标：让业务员知道为什么先看这些公司。

功能点：

- 综合评分
- 子分明细
- 推荐理由生成
- 结果排序切换
- 人工反馈回流

推荐理由示例：

- 该公司位于德国，与你输入国家一致
- 官网内容与 `laser cutting` 高相关
- 最近 12 个月存在相关进口记录
- 找到采购经理级联系人和企业邮箱

首版验收标准：

- 每个结果至少显示 2 到 4 条推荐理由
- 用户可按综合分、海关活跃度、联系人完整度排序

### 20.9 结果管理模块

目标：让业务员对查询结果进行后续处理。

功能点：

- 收藏公司
- 批量勾选
- 导出 Excel/CSV
- 添加备注
- 标记无效线索
- 标记高价值线索

首版验收标准：

- 支持单条与批量导出
- 支持结果反馈，供后续优化排序

## 21. 数据主档整理方案

本节聚焦如何把现有分散数据整理成 Agent 可调用的数据底座。

### 21.1 数据治理目标

目标不是一次性做完全部数据治理，而是优先完成会直接影响首版检索效果的部分：

- 统一公司实体
- 统一联系人实体
- 打通公司与海关记录
- 建立来源追踪与置信度体系

### 21.2 建议的数据分层

建议将数据体系拆为 5 层：

- `ODS`：原始接入层，保留所有来源原始数据
- `DWD`：清洗标准层，完成字段归一和标准化
- `DWM`：主题映射层，建立公司、联系人、海关之间的关系
- `DIM`：主档维表层，沉淀统一主档
- `ADS`：应用服务层，为搜索和 Agent 提供查询接口

### 21.3 原始接入层设计

要求：

- 不改原始数据结构
- 每条数据带上来源系统、抓取时间、原始主键
- 保留原始名称、原始国家、原始网址、原始联系人字段

目的：

- 便于回溯问题
- 便于后续重跑清洗规则
- 避免因早期治理误伤原始数据

### 21.4 标准化层设计

标准化工作建议包括：

- 公司名称清洗
- 域名提取与标准化
- 国家代码统一
- 城市字段归一化
- 联系人姓名切分
- 职位标签归一化
- 邮箱格式校验
- HS Code 格式统一

建议建立统一字典：

- 国家字典
- 行业字典
- 岗位字典
- 海关产品映射字典
- 公司规模字典

### 21.5 实体识别与映射层设计

这是整个项目的关键中间层。

建议建立以下映射规则：

- `company_raw -> company_master`
- `contact_raw -> contact_master`
- `customs_raw -> customs_record`
- `domain -> company_master`
- `contact -> company_master`

公司映射建议权重：

- 域名完全一致：高权重
- 标准名称高相似：高权重
- 国家一致：中权重
- 城市一致：辅助权重
- 海关主体近似：中权重

联系人映射建议权重：

- 邮箱完全一致：高权重
- 姓名 + 公司一致：高权重
- LinkedIn 链接一致：高权重
- 姓名近似 + 域名一致：中权重

### 21.6 主档层设计

主档层不只是表结构，更是一套“谁是最终可信记录”的决策机制。

建议原则：

- 一个公司只有一个主记录 ID
- 一个联系人只有一个主记录 ID
- 同时保留所有来源记录与映射关系
- 主档字段可按来源优先级自动覆盖
- 对冲突字段保留人工校正能力

字段覆盖优先级示例：

- 官网域名：优先可信来源或人工确认结果
- 公司标准名：优先人工确认，其次高质量结构化来源
- 联系人职位：优先更新时间近的来源
- 邮箱：优先企业域名一致的邮箱

### 21.7 置信度体系设计

建议为公司、联系人、邮箱、海关关联分别建立置信度。

置信度等级可分：

- `A`：高可信，可直接给业务员优先使用
- `B`：中可信，可展示但需谨慎
- `C`：低可信，仅作为辅助参考

示例：

- 公司域名与标准名高度一致，多个来源同时出现，可评 `A`
- 联系人只有名字和职位，没有邮箱，可评 `B`
- 海关主体名模糊匹配且无域名佐证，可评 `C`

### 21.8 人工校正机制

首版不要追求百分百自动化，建议保留运营或数据团队人工校正能力。

建议支持：

- 手动合并公司
- 手动拆分误合并公司
- 手动绑定海关主体与公司
- 手动确认联系人归属
- 手动提升或降低可信度

### 21.9 数据治理优先顺序

建议按以下顺序推进：

1. 公司主档
2. 域名标准化
3. 联系人与公司关系
4. 海关主体映射
5. 联系人去重与邮箱置信度
6. 行业与岗位标准字典

## 22. Agent 一次查询的完整工作流

本节从用户发起查询到结果返回，描述一次完整运行过程。

### 22.1 用户视角流程

1. 业务员输入自然语言需求
2. 系统展示解析出的结构化条件
3. 用户确认或调整条件
4. 系统开始检索
5. 结果页展示推荐公司、联系人、邮箱、海关摘要
6. 用户收藏、导出或标记结果

### 22.2 系统内部流程

#### 第一步：意图解析

输入：

- 自然语言请求

输出：

- 产品关键词
- 国家/地区
- 联系人偏好
- 海关限制条件
- 公司规模等辅助条件

#### 第二步：查询改写

系统将用户输入转换为多个检索任务：

- 产品词扩展任务
- 公司召回任务
- 海关匹配任务
- 联系人补全任务

#### 第三步：候选公司召回

系统从多个数据源并行召回：

- 公司基础库
- 域名/官网库
- 行业标签库
- 海关映射库

#### 第四步：候选公司去重合并

系统将不同来源中的相似公司归并为统一公司实体，并保留来源明细。

#### 第五步：联系人与邮箱补全

针对候选公司补全：

- 关键联系人
- 优先职位
- 企业邮箱
- 联系人可信度

#### 第六步：海关活跃度计算

针对候选公司匹配对应海关记录，并生成：

- 最近交易时间
- 最近周期内交易频次
- HS Code 匹配度
- 贸易活跃标签

#### 第七步：综合评分

系统计算：

- 产品匹配分
- 海关分
- 联系人完整分
- 数据可信分
- 综合推荐分

#### 第八步：推荐解释生成

系统为每个结果生成结构化解释和自然语言解释。

#### 第九步：交互反馈记录

记录用户行为：

- 哪些公司被收藏
- 哪些结果被导出
- 哪些线索被标为无效

这些反馈用于后续排序优化。

### 22.3 异常处理流程

如果某次查询遇到问题，建议按以下方式处理：

- 没有找到联系人：仍返回公司，并标记“待补全联系人”
- 没有海关记录：仍返回公司，但不打海关活跃标签
- 存在重复公司：展示主记录并允许查看来源明细
- 结果可信度低：显式标注为低置信度

### 22.4 查询输出结构建议

单条结果建议包含：

- 公司主信息
- 官网与域名
- 行业与地区
- 海关摘要
- 联系人列表
- 邮箱与可信度
- 推荐理由
- 数据来源数
- 综合评分

## 23. 技术架构建议

首版架构建议采用“数据服务化 + 检索编排 + Agent 解释层”的模式。

### 23.1 总体架构分层

- 数据接入层
- 数据治理层
- 检索服务层
- Agent 编排层
- 应用交互层

### 23.2 数据接入层

职责：

- 接入现有多个库表和文件数据
- 建立定时同步机制
- 保留原始来源信息

建议能力：

- 批量导入
- 定时同步
- 增量更新
- 原始数据版本记录

### 23.3 数据治理层

职责：

- 标准化、去重、实体映射、主档生成

建议拆分服务：

- 公司标准化服务
- 域名解析服务
- 联系人标准化服务
- 海关映射服务
- 主档生成服务

### 23.4 检索服务层

职责：

- 为前台查询提供统一搜索 API

建议拆分能力：

- 公司搜索服务
- 联系人搜索服务
- 海关过滤服务
- 排序评分服务
- 导出服务

### 23.5 Agent 编排层

职责：

- 解析自然语言
- 生成查询计划
- 调用检索服务
- 组装结果解释

Agent 不应直接操作底层原始数据，而应通过标准服务接口访问。

建议在这一层控制：

- Prompt 模板
- 工具调用策略
- 查询条件校验
- 结果解释格式

### 23.6 应用交互层

职责：

- 提供业务员操作界面
- 展示结果、详情、推荐理由、导出能力

建议页面：

- 搜索首页
- 结果列表页
- 公司详情抽屉或侧栏
- 收藏与导出页
- 查询历史页

### 23.7 日志与监控

建议从首版开始建设：

- 查询日志
- 工具调用日志
- 数据命中率监控
- 查询耗时监控
- 导出与收藏行为监控
- 低可信结果占比监控

### 23.8 权限建议

虽然是内部系统，但仍建议做基础权限：

- 业务员：查询、收藏、导出
- 主管：查看团队使用情况
- 数据运营：人工校正主档
- 管理员：数据源配置、规则维护

## 27. 验收标准建议

建议把验收拆成业务验收和系统验收。

### 27.1 业务验收

- 业务员能在一次查询中拿到一批可开发公司
- 结果中有足够比例附带联系人和企业邮箱
- 结果排序符合业务直觉
- 推荐理由能被业务员理解和接受

### 27.2 系统验收

- 查询成功率稳定
- 首批结果返回时间可接受
- 导出流程稳定
- 主档合并规则可追溯
- 低可信结果有明确标识

## 29. 当前建议结论

对于你们当前阶段，最重要的不是立刻做一个“全自动 AI 销售员”，而是先把现有数据资产整理成一个业务员真正能用的线索发现系统。

因此，推荐下一步执行顺序是：

1. 盘点数据源和字段
2. 建公司与联系人主档
3. 建海关映射规则
4. 做统一检索和结果页
5. 再接入 Agent 解释与自然语言查询

这样最稳，也最有机会在较短时间内看到真实业务价值。
